Include and exclude filters. Filter recommendations to or away from specific catalog fields: exclude Clearance, include in_stock = true, exclude watched = true. The filters are where taste lives.
Item sequencing. You decide which of the recommended items each message serves: the first, the top three, and so on.
Location matching. The product location catalog carries lat/long per item; campaigns can filter picks to a range around the user's last known location. Distance handled inside CleverTap.

Setup checklist (hand this to the team)

  1. Export the catalog as CSV with a comma-separated header row; every subsequent row is one item. For location plays, use the product location catalog variant with lat/long columns.
  2. Map catalog columns to event properties. Multiple columns can map to one event's properties, but all mapped properties must belong to the same event. Values read as strings.
  3. Schedule the catalog sync to match how fast the underlying data moves: daily for listings and menus, per-cycle for tariffs and rate sheets.
  4. Create recommendations at least 24 hours before campaign launch; the engine takes up to a day to publish.
  5. Add exclude filters for terminal states (sold, filled, watched, completed) so the engine never recommends something the user cannot act on.
  6. For PBS campaigns, remember the 60-day basis-event rule when reconciling segment size against messages served.

Honest edges

Recommendations need behavioural volume: thin B2B event counts can starve the engine, so validate co-view signal before trusting picks (see logi-01). Catalog freshness is your upload cadence, nothing more; minute-fresh data belongs to Linked Content in Field Guide 06. ⚙️ 1 · Catalogs: upload & mapping CSV in, event mapping on▶

Part A · Prepare the CSV

One header row, one item per row
Comma-separated header names the fields; every subsequent row is one catalog item. Keep an unambiguous item ID column that also exists as an event property (e.g. item_id on Product Viewed).
Pick the catalog variant
Product catalog for items; product location catalog when items carry lat/long columns, which unlocks distance-range filtering at serve time.
Do the maths before upload
Catalog values are read as strings regardless of uploaded type. Discounted prices, EMI amounts, savings percentages: compute them server-side and upload as columns. Liquid compares and renders; it does not calculate.

Part B · Upload via API (for scheduled syncs)

Request a pre-signed URL
POST to /get_catalog_url on your region’s base URL with the standard account headers. Most SEA accounts sit on sg1.api.clevertap.com; confirm from the dashboard login URL.
curl -X POST https://sg1.api.clevertap.com/get_catalog_url \ -H 'Content-Type: application/json' \ -H 'X-CleverTap-Account-Id: ACCOUNT_ID' \ -H 'X-CleverTap-Passcode: PASSCODE'
PUT the CSV to the pre-signed URL
The URL is valid for a maximum of 24 hours; single-upload cap is 5 GB.
curl --request PUT --upload-file "catalog.csv" "<PRE-SIGNED URL>"
Signal completion
A second API call (same Catalog API doc page) tells CleverTap the upload is done; the catalog then appears on the dashboard. Schedule the whole loop at the cadence the data moves: daily for listings and menus, per-cycle for tariffs and rate sheets.

Part C · Map columns to events

All mapped columns → one event
Catalog columns map to event properties, and every mapped column must belong to the same event. Choose the event that carries your item ID at the highest honest volume; that mapping feeds both Catalog send-time lookups and the Rec Engine’s learning.
🧠 2 · Recommendations: create & publish basis · timeframe · filters · 24h▶
Choose the basis event
Recommendations generate from events mapped to the catalog: items Viewed Together or Searched Together within your chosen timeframe (last 10 days, 30 days, and so on).
Add include/exclude filters
Filter picks to or away from catalog fields: exclude category = Clearance, include in_stock = true, exclude watched = true. The filters are where taste lives — and where terminal states (sold, filled, completed) get fenced off so the engine never recommends something the user cannot act on.
Build it the day before
The data science engine takes up to 24 hours to publish a recommendation after creation. Create it the day before launch, not the morning of. This is the single most common go-live surprise.
Know the PBS 60-day rule
In Past Behavior Segment campaigns, users must have performed the recommendation’s basis event within the last 60 days to receive the campaign. Segment size can legitimately exceed messages served; when a client asks why, this is why.
✉️ 3 · Serving in campaigns picker · sequencing · Catalog STP Liquid▶
Pick the published rec + item sequence
In the campaign, choose the published recommendation, the trigger event, and which of the recommended items each message serves: the first, the top three, and so on. Location-catalog plays can additionally filter to a range around the user’s last known location — distance handled inside CleverTap.
Catalog send-time lookups, no engine required
The second mechanism in this guide: a send-time lookup of the catalog row matched to the user’s event. Always carry a fallback.
{{ Catalog.["catalog_name"].price | default:"see app for price" }} {{ Catalog.["fare_sheet"].route_name | default:"your route" }}
QA on the live channel
CleverTap does not render recommendations in Web Pop-up campaign preview. Test on the served channel, not the preview pane. Send-to-self on push and email before any client demo.
🛡️ 4 · Guardrails & TAM talking points the difference between demo and deployment▶
Freshness = upload cadence. Catalog data is exactly as fresh as the last sync, nothing more. Minute-fresh data (live stock, live prices from an endpoint) belongs to Linked Content — Field Guide 06.
Thin data starves the engine. Recommendations need behavioural volume. Low-traffic B2B or niche catalogs may not form co-view relationships; validate signal before trusting picks (see logi-01 in the gallery).
Defaults everywhere. Every Catalog lookup in production carries a default:. A blank price in a client’s message is a credibility incident, not a bug.
Strings, always. Repeat of the big one: catalog values are strings. Arithmetic happens server-side, before upload.
Unpublished edges. Per-account catalog count limits, max rows per catalog, and rec-item caps per channel are not published in CleverTap documentation as of July 2026. Treat as unconfirmed and check with your CSM before designing around them. That CSM is us, which is rather the point.
